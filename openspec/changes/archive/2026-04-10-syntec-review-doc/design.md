## 背景

`/syntec:review-doc` 是一個 Claude Code custom command（`.claude/commands/syntec/review-doc.md`），用於 review 設計文件。實際使用情境中，一次 review 通常包含 1 份主文件加上 1~2 份補充文件，補充文件由實作者自由撰寫，沒有固定格式。Command 需要能處理任意文件結構。

## 目標與非目標

**目標：**
- 透過 Atlassian MCP 讀取所有 Confluence 頁面，不需要使用者手動貼內容
- 以語意判斷略過行政資訊，不依賴硬編碼的章節名稱
- 各文件各章節獨立 review，輸出分級問題清單
- 跨文件識別「規格/需求」與「測試/驗收」內容，比對覆蓋度

**非目標：**
- 不自動修改文件
- 不追蹤外部連結（文件中引用的外部 URL 一律略過）
- 不支援非 syntecclub.atlassian.net 的 Confluence 實例
- 不儲存 review 結果（純對話輸出）

## 設計決策

**語意略過而非硬編碼章節名稱**
原始設計假設文件符合 R-RD13-00-01 v6 模板，可以硬編碼略過清單。實際上補充文件沒有固定格式，硬編碼章節名稱會失效。改為在 command prompt 中提供語意提示，讓 Claude 自行判斷哪些內容屬於「行政資訊」（作者姓名、日期、ISSUE 編號、簽核欄位、SOP 說明等）並略過。

**略過提示寫在 command 檔案中**
提示就是 prompt 的一部分，寫在 `review-doc.md` 裡。未來若需要調整略過規則，直接修改該檔案即可，不需要額外設定機制。

**多文件輸入：所有 URL 平等對待**
使用者在呼叫時提供主文件 URL，補充 URL 從對話 context 中偵測。所有文件讀取後合併處理，review 時按「文件 → 章節」分組輸出，跨文件比對則統一從所有文件中尋找規格與測試內容。

**跨文件覆蓋度比對以語意為準**
不假設特定章節名稱，而是由 Claude 從所有文件中識別「在描述規格/需求的段落」和「在描述測試案例/驗收條件的段落」，然後比對覆蓋度。比對結果作為獨立的「跨文件比對」段落輸出。

**硬編碼 cloudId = syntecclub.atlassian.net**
這是公司唯一的 Confluence 實例，不需要動態配置。

## 風險與取捨

- 【風險】語意判斷略過可能偶爾誤判（把有意義的章節當行政資訊略過）→ 對策：提示清單設計保守，只提示明確的行政資訊類型；誤判比漏掉問題的傷害小
- 【風險】跨文件規格/測試識別在文件結構差異大時可能不準確 → 對策：輸出時說明 Claude 識別到的規格來源與測試來源，讓使用者能驗證
- 【風險】MCP tool 呼叫失敗（網路、權限）→ 對策：失敗時告知使用者並停止

### 需求：Command 接受一或多個 Confluence URL
`/syntec:review-doc` command 必須接受至少一個 Confluence URL 作為主文件。Command 必須同時從當前對話 context 中偵測使用者提供的額外 Confluence URL，一併讀取作為補充文件。所有文件透過 Atlassian MCP 工具 `getConfluencePage`（cloudId: `syntecclub.atlassian.net`）取得。

#### 情境：只有主文件
- **當** 使用者執行 `/syntec:review-doc <url>`，對話中沒有其他 Confluence URL
- **則** command 只讀取主文件並進行 review

#### 情境：主文件加補充文件
- **當** 使用者執行 `/syntec:review-doc <url>`，且對話中另有一或多個 Confluence URL
- **則** command 讀取所有文件，review 時涵蓋全部文件

#### 情境：MCP 取得失敗
- **當** 任一文件的 `getConfluencePage` 回傳錯誤或空內容
- **則** command 必須向使用者回報哪份文件失敗，並停止執行

### 需求：以語意判斷略過行政資訊
Command 不得依賴硬編碼的章節名稱來決定略過哪些內容。必須以語意判斷，略過以下性質的內容：
- 純行政資訊：作者姓名、日期、ISSUE 編號、簽核欄位
- 審查流程說明（如「審查後請做 X」、「自我 review SOP」）
- 無實質技術內容的範本填寫說明或佔位文字

#### 情境：章節內容為純行政資訊
- **當** 某章節只包含作者、日期、簽核等行政欄位
- **則** command 略過該章節，不產生 review 輸出

#### 情境：章節包含實質技術內容
- **當** 某章節包含需求描述、架構說明、測試案例等技術內容
- **則** 無論章節名稱為何，command 必須 review 該章節

### 需求：各章節 review 輸出分級問題
Review 輸出中的每個問題必須以以下三種嚴重程度 tag 之一作為前綴：
- `[嚴重]`：邏輯矛盾、前後不一致，或會導致實作或測試錯誤的內容
- `[中等]`：描述模糊、未定義邊界條件，或缺少重要細節
- `[輕微]`：錯字、格式問題，或用語不一致

#### 情境：問題為邏輯矛盾
- **當** 文件中兩個陳述互相矛盾
- **則** 該問題必須標記為 `[嚴重]`

#### 情境：問題為缺少邊界條件
- **當** 某個行為有描述但其邊界或錯誤情況未說明
- **則** 該問題必須標記為 `[中等]`

#### 情境：問題為錯字或格式問題
- **當** 有單字拼錯或格式不一致
- **則** 該問題必須標記為 `[輕微]`

### 需求：輸出按文件與章節分組
Review 輸出必須按「文件 → 章節」的層級分組。每份文件以文件標題作為群組標題，其下各章節再依序列出問題。

#### 情境：多份文件各有問題
- **當** 主文件和補充文件都發現問題
- **則** 兩份文件的問題分別在各自的群組下輸出，不混合

#### 情境：章節沒有問題
- **當** 某章節 review 後沒有發現問題
- **則** 該章節可從輸出中省略，或標記為「無問題」

### 需求：跨文件比對規格與測試覆蓋度
完成各章節 review 後，command 必須從所有文件中以語意識別「規格/需求」性質的內容與「測試/驗收」性質的內容，並比對覆蓋度。輸出時必須說明識別到的規格來源與測試來源，讓使用者能驗證。比對結果以獨立的「跨文件比對」段落呈現。

#### 情境：規格內容缺乏測試覆蓋
- **當** 某段規格描述的行為在所有文件中找不到對應的測試案例
- **則** 在「跨文件比對」段落中回報，標記為 `[嚴重]`

#### 情境：找不到規格或測試內容
- **當** 所有文件中都無法識別出規格性質或測試性質的內容
- **則** command 必須說明無法進行跨文件比對，並描述原因

#### 情境：完整覆蓋
- **當** 所有識別到的規格行為都有對應測試案例
- **則** 跨文件比對段落回報無問題

### 需求：略過文件中的外部連結
Command 不得追蹤或取得 Confluence 文件內容中找到的任何外部 URL。

#### 情境：文件包含外部超連結
- **當** 文件內文包含指向外部網站或其他系統的 URL
- **則** command 必須忽略這些 URL，不嘗試取得它們

Review 所有提供的 Confluence 設計文件，輸出結構化的問題清單。

## 輸入

第一個參數是主文件的 Confluence URL。同時掃描當前對話 context，偵測使用者貼的其他 Confluence URL，一併作為補充文件讀取。

## 讀取文件

使用 Atlassian MCP 工具 `getConfluencePage`（cloudId: `syntecclub.atlassian.net`）讀取所有文件。

若任一文件讀取失敗，回報哪份文件失敗並停止執行。

## 略過規則

**不依賴章節名稱**，以內容性質判斷是否略過。略過以下類型的內容：

- 純行政資訊：作者姓名、日期、ISSUE 編號、簽核欄位、審查人清單
- 流程說明：「審查後請做 X」、「自我 review SOP」、「審查後 SOP」等操作步驟
- 無實質技術內容的範本佔位文字（如「請在此填寫…」）

若某章節混合了行政資訊與技術內容，只略過行政部分，技術內容仍須 review。

## Review 規則

對每份文件的每個非略過章節進行 review，找出以下問題並標記嚴重程度：

- **`[嚴重]`**：邏輯矛盾、前後不一致、缺少關鍵資訊，會導致實作或測試錯誤
- **`[中等]`**：描述模糊、邊界條件未說明、重要細節缺漏，需要補充才能正確實作
- **`[輕微]`**：錯字、格式不一致、用語不統一，不影響理解但應修正

外部連結（非 Confluence 的 URL）一律不追蹤、不讀取。

## 輸出格式

輸出按「文件 → 章節」分組：

```
## 文件：<文件標題>

### <章節名稱>
- [嚴重] <問題描述>
- [中等] <問題描述>

### <章節名稱>
- [輕微] <問題描述>

## 文件：<補充文件標題>

### <章節名稱>
- [中等] <問題描述>
```

沒有問題的章節可省略，或標記「無問題」。

## 跨文件比對

完成各章節 review 後，從所有文件中以語意識別：

- **規格/需求內容**：描述系統應該做什麼、功能行為、輸入輸出、限制條件
- **測試/驗收內容**：測試案例、驗收條件、測試情境

比對規格內容是否有對應的測試案例覆蓋。輸出時說明識別到的規格來源與測試來源（讓使用者能驗證識別是否正確）。

```
## 跨文件比對

**規格來源**：<文件A> § <章節>、<文件B> § <章節>
**測試來源**：<文件A> § <章節>

### 未覆蓋的規格
- [嚴重] <規格描述> — 在所有文件的測試章節中找不到對應測試案例

### 結論
<整體覆蓋度評估>
```

若所有文件中都找不到規格性質或測試性質的內容，說明無法進行跨文件比對並描述原因。
